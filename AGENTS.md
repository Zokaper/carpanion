# Developer Guide (AGENTS.md)

Carpanion is a Flutter **car dashboard / infotainment head-unit** app for an Android
tablet mounted in a vehicle. It started as a speedometer and grew into a full
infotainment hub: driving metrics, maps/nav automation, media control, notifications,
system controls, a **Collaborative Playback** feature (passengers add/edit the queue
from their phones via a PWA), and a **native music queue** driven off YouTube Music
(V4: plus the user's **personalized YT Music mixes** — Supermix etc. — via an in-app login).

This is the **standing project reference** — shared, tool-agnostic context for whichever
agent (Claude Code, Codex, or a human) is picking the project up. Read it first when
picking the project up on a new machine.

**Companion files:**
- **`STATUS.md`** — recent/active work log. Append an entry when you finish a chunk of
  work; check it at the start of a session to see what the other agent/session last did.
- **`STATUS_ARCHIVE.md`** — older `STATUS.md` entries, moved out to keep `STATUS.md` short.
- This file (`AGENTS.md`) is for **standing** info (architecture, how things work, known
  risks) — not for "what did I do today" (that's `STATUS.md`). Update it when the standing
  facts change (a new subsystem, a semantics change), not for routine task notes.

## 🛠️ Build & Run

Uses **`puro`** for Flutter version management — always prefix Flutter commands with it.

* **Run**: `puro flutter run` (add `-d <device-id>` for a specific device)
* **Hot reload / restart**: `r` / `R` in the run terminal
* **Deps**: `puro flutter pub get`   •   **Clean**: `puro flutter clean`
* **Verify a change compiles (Dart + Kotlin)**: `puro flutter build apk --debug`
* **Analyze**: `puro flutter analyze` — NOTE: `lib/main_corrupted.dart` is a dead,
  broken backup file that throws ~30 analyzer errors; ignore it (it is not imported/
  compiled). There are also many pre-existing `withOpacity` deprecation **info** lints —
  those are noise, not errors.

### Backend (PWA + WebSocket relay)
* **Path**: `backend/`   •   **Start**: `node server.js`
* **Hosted**: the dashboard app and passenger phones both connect to the **remote**
  backend at `https://carpanion.onrender.com` (hardcoded in `collab_service.dart` and
  `queue_tab.dart`). Render **auto-deploys `backend/` on push to `main`**. So PWA/server
  changes only reach phones after a push to main; Flutter/native changes need an app rebuild.

### Git workflow
* Active branch is **`V4`** (off `V3.5`); `V3.5` and `main` both track the last shipped release.
  V4 is **in progress** — do NOT push it until asked. When you do push a release, push to BOTH:
  `git push origin V4`, then fast-forward `main` (`git checkout main && git merge --ff-only V4 && git push origin main && git checkout V4`); also update `V3.5` if it should track.
  (The older `V3` branch is frozen at the V3 release — do **not** push to it anymore.)
* When staging, **exclude** `macos/Flutter/GeneratedPluginRegistrant.swift` (regenerated by
  `pub get`, not real work).

## 🧪 Local testing (laptop, no phone)

Full runbook is in **`TESTING.md`**. In short — you can exercise the app on this machine
without the real phone, using the Android emulator + a local backend:

* **Backend + PWA**: `node server.js` in `backend/` (or `scripts/start-backend.ps1`) — port
  3000, serves the socket.io relay **and** the passenger PWA. Node LTS required.
* **Dashboard**: run on the **`S25plus` emulator** (an AVD built to match the real device —
  see below), pointed at the local backend:
  `puro flutter run --dart-define=BACKEND_URL=http://10.0.2.2:3000`
  (or `scripts/run-dashboard.ps1`). `10.0.2.2` = host localhost from inside the emulator.
* **Test-only dart-define**: `BACKEND_URL` overrides the hardcoded backend (`collab_service.dart`);
  defaults to prod, safe to keep. A dev-only `network_security_config.xml` allows cleartext to
  10.0.2.2/localhost. *(The old `DEV_BYPASS_AUTH` flag is gone — the Collab tab no longer has a
  Google-sign-in gate, so no bypass is needed; sign-in is optional, in Settings, for demo search.)*
* **Passenger PWA**: open `http://localhost:3000/?session=<code>` (session code is on the
  dashboard's COLLAB tab).
* **Can't test without the real phone**: actual YT Music audio, auto-advance on song-end, and
  now-playing highlight (native MediaSession polling is empty on the emulator).

### 📱 Target device & the `S25plus` emulator
The app runs on a **Samsung Galaxy S25+ phone in landscape** (not a tablet, despite the
"head-unit" framing; it forces landscape). **Layout must be validated at that device's size** —
an AVD named **`S25plus`** exists for this: **1080×2340 @ 450 dpi** (landscape 2340×1080,
~832×384 dp). `scripts/run-dashboard.ps1` defaults to it. Do NOT judge layout on a default
phone AVD — a small screen produces false overflow warnings. Recreation steps + the exact-match
caveat are in `TESTING.md`.

### ✅ How to test (preference)
**Validate what changed — do not sweep the whole app every time.**
* Changed logic (e.g. favorites, queue, collab) → drive that specific flow on the emulator and
  confirm it works (assert via the run log, backend log, or on-screen state).
* Changed a tab's layout → open that tab, screenshot it at S25+ resolution, and confirm it
  looks right. Flutter prints `A RenderFlex overflowed by N pixels` to the run log — scrape it.

## 🏗️ Architecture Overview

Three cooperating layers:

1. **Flutter UI + state** (`lib/`) — dashboard, media panel, favorites, collab/queue tab.
   State lives in `ChangeNotifier` providers wired in `main.dart`'s `MultiProvider`:
   `DashboardProvider`, `DynamicThemeProvider`, `YouTubeService`, `CollabService`
   (the last is created after the others so it can `context.read` them).
2. **Native Android** (`android/.../MainActivity.kt` + `DashcamListenerService.kt`) —
   platform channel `com.example.car_dashboard/system`; drives YT Music via its
   MediaSession, reads media metadata, dashcam/brightness/ringer/network, notification
   listening.
3. **Node backend + PWA** (`backend/`) — `server.js` is a socket.io **relay** between the
   dashboard and passenger phones; `backend/public/` is the passenger PWA
   (`index.html`, `app.js`, `sw.js`). Passengers join by scanning a QR shown on the tablet.

## 🎵 The Core Idea: one native "Now Playing Queue"

The pivotal capability: we can drive YouTube Music **without flashing it to the
foreground** by sending commands to its MediaSession. Native handler
**`playFromMediaSession`** (`MainActivity.kt`) calls `transportControls.playFromUri`
with `https://music.youtube.com/watch?v=<videoId>` — confirmed to play silently in the
background (`playFromSearch` is silently ignored by YT Music, so we always use URI).

On top of that we built a **single app-managed queue** — the app owns the list, plays it
one track at a time via `playFromMediaSession`, and auto-advances. YT Music is just the
audio backend. This queue powers **both** Collab (passenger adds) and **Favorites**
(albums/artists/songs). Everything funnels through **`CollabService`**.

### `CollabService` (`lib/services/collab_service.dart`) — the playback engine
App-lifetime service (survives navigation; the collab/queue tab is a thin view over it).
- **Queue** lives in `YouTubeService.currentQueue` (a `List<Map>` of `{id, videoId,
  originalVideoId, title, artist, thumbnail}`), persisted to `SharedPreferences`
  (`collab_queue`).
- **`playAt(index)`** → sets `_currentPlayingIndex`, `_playbackActive = true`, invokes
  `playFromMediaSession`; falls back to a launch intent if the MediaSession is unreachable.
- **Auto-DJ** (`_onDashboardUpdate`, listens to `DashboardProvider`): when the playing
  track (from native polling) leaves the queue (song ended → YT autoplay), it advances to
  the next queue item. **Gated on `_playbackActive`, NOT on collab being enabled** — so
  favorites playback auto-advances even with Collab off. It also re-syncs the highlighted
  index by fuzzy title-match and emits `update_playing_status` / `update_play_state`.
- **Favorites entry points**: `playFavoriteSong({videoId,title,artist,thumbnail})` and
  `loadQueueAndPlay(List<songs>)` — both **replace** the queue and play from 0.
- **Persistence** (`SharedPreferences`): `collab_session_id`, `collab_enabled`,
  `collab_index`. On restart it does a **silent restore**: re-selects the saved song
  (matching what YT Music is already playing) but does NOT replay it (would restart the
  song at 0:00); resumes Auto-DJ if a match is found.
- **Media controls**: `next()`/`previous()` (queue-aware, else native prev/next).

### Sharing-layer Collab model (IMPORTANT semantics)
There is ONE queue. **`_enabled` = "passenger sharing is open", NOT playback.**
- Toggling Collab ON/OFF **never** starts, stops, or clears playback — flip it mid-album
  and the music is untouched. Enabling just opens the door for passengers.
- Passenger **write** actions (`passenger_*` socket handlers) are gated on `_enabled`
  (and the per-permission toggles `_allowEditing` / `_allowMediaControl`).
- Playback starts from: a favorite (song/album/artist), tap-to-play a queue row, or a
  passenger add to an idle queue (`_maybeAutoStart`). **Enabling Collab does NOT auto-play.**
- `newSession()` (the only full reset: new session id + clear queue) and `clearQueue()`
  (empties queue, keeps session) both reset `_playbackActive`/index.

## 🔎 YouTube Music integration (`lib/services/youtube_service.dart`)

- **OAuth** via `google_sign_in` (youtube scope) → `YouTubeApi` (Data API v3). **Optional /
  on-demand** — it powers ONLY the passenger "search YouTube for demos" toggle (`searchSongs`,
  Data API search of all of YouTube, incl. unreleased/bootleg uploads not in the YT Music catalog)
  and a title-lookup fallback for bare shared links (`_fetchYouTubeVideoTitle`). Both degrade
  gracefully when signed out (`_youtubeApi == null` → `[]`/`null`). There is **no sign-in gate**;
  sign-in lives in **Settings → Media & Collab** (`settings_screen.dart`, the `_GoogleAccountCard`).
  Default passenger search, playback, queue ops and all Favorites are anonymous (Innertube +
  MediaSession) and need no OAuth. No YT-playlist API writes remain (`addVideoToPlaylist` only
  appends to the local queue despite its legacy name).
- **Innertube (WEB_REMIX, anonymous)** — the good stuff. Public API key, POST to
  `music.youtube.com/youtubei/v1/...`:
  - **`searchYTMusicSongs(query, {limit})`** — "Songs" filter
    (`params: 'EgWKAQIIAWoKEAkQBRAKEAMQBA=='`). Returns ranked `{videoId,title,artist,
    thumbnail}` list via `_parseSongRenderer`. This is the authoritative song resolver AND
    the source of the PWA's search results (replaced iTunes, which ranked badly).
  - **`getAlbumTracks(album, artist)`** — `search` → find `MPRE…` album browseId →
    `browse` → collect `musicResponsiveListItemRenderer` tracks (fills missing artist/
    thumbnail from album context).
  - **`getArtistRadioTracks(artist)`** — `search` → `UC…` artist browseId → `browse` →
    find an `RD…` radio playlistId → `next` → parse `playlistPanelVideoRenderer`. Gives
    YT Music's own popularity-weighted shuffle order.
  - Generic recursive JSON walkers: `_findBrowseId`, `_findRadioPlaylistId`,
    `_findFirstThumbnail`, `_collectByKey`, `_runsText`; shared POST via `_innertubePost`.
  - **All fetchers return `[]` on failure** so callers fall back to the legacy YT Music
    launch. They have debug logging (`getAlbumTracks: … → N tracks`, etc.).
- **`replaceQueue(songs)`** / **`addResolvedSong(...)`** / `addVideoToPlaylist` / dedup +
  5s repeat guard, all persisting the queue.

> ⚠️ The album/artist Innertube parsing was written **without live-response validation** —
> it's the main risk area. If album/artist playback falls back to the YT flash, check the
> debug logs and adjust the parse. Search + single-song resolution are proven/reliable.

## ⭐ Favorites (typed) — `main.dart` `DashboardProvider` + `_MediaControlPanelState` + `_FavoritesSidebarState`

Favorites are now **typed** maps: `type ∈ {song, album, artist, playlist}`.
- `song` → stores resolved `videoId` + `thumbnail` (resolved via `searchYTMusicSongs` at
  favorite time so playback is instant).
- `album`/`artist` → store name(s); tracks are fetched on play.
- `playlist` → **retired from the favorites list in V4** (see the mix grid below). The 3 seeded
  `playlist` favorites were removed and `loadFavorites` one-time strips any `type=='playlist'`
  from `fav_json`; the favorites list now shows **song/album/artist only**. `_launchPlaylist`
  still exists for any legacy launch path.
- **Persistence**: single JSON list under `fav_json` (migrates the old parallel
  `fav_titles/urls/subtitles` keys on first load; `type` defaults to `playlist`).
- **Media-player buttons** (top-right of the now-playing panel, `_MediaControlPanelState`):
  heart = song (`_toggleSongFavorite` resolves the id), `Icons.album` = album (needs
  `_currentAlbum`), `Icons.interpreter_mode` = artist. Toggles via `DashboardProvider`
  helpers `isSongFavorited/isAlbumFavorited/isArtistFavorited`, `addSongFavorite`,
  `toggleAlbumFavorite`, `toggleArtistFavorite`.
- **`_currentAlbum`** is populated by polling the native **`getCurrentMediaMetadata`**
  (title/artist/album from the YT Music MediaSession) in `DashboardProvider._startMediaPolling`
  — the media plugin only exposes track/artist.
- **Tapping a favorite** → `_playFavorite` dispatcher: song→`playFavoriteSong`;
  album→`getAlbumTracks`→`loadQueueAndPlay`; artist→`getArtistRadioTracks`→
  `loadQueueAndPlay`; playlist / any failed fetch → `_launchPlaylist` (legacy YT launch).

## 🎧 V4: Personalized YT Music mixes (Supermix grid) — **on-device VERIFIED (login + feed), one bug open**

V4's headline feature: play the user's OWN personalized YT Music mixes (Supermix, Quick Picks,
Discover Mix, …) **in our native queue**, not just launch the YT Music app. It required
authenticating to YT Music (the anonymous Innertube client has no user identity).

**Status (verified on the real S25+ 2026-07):** in-app Google login worked first try (smooth, no
"browser not secure" block); the 4 mix tiles populated immediately on the favorites screen; tapping
Supermix filled the native queue and played. **✅ Audio+video duplicate bug FIXED:**
`_playlistPanelTracks` now dedups by title+artist, keeping the first occurrence in place but
preferring the audio (`ATV`) videoId over its OMV/UGC video twin (via each renderer's
`navigationEndpoint…watchEndpointMusicConfig.musicVideoType`). *(Code fix in; still worth an
on-device re-check that Supermix no longer double-queues.)*

### Auth — cookie/SAPISIDHASH login (the SimpMusic/ViMusic approach, NOT OAuth device flow)
- **`lib/ui/ytmusic_login_webview.dart`** — full-screen `webview_flutter` login (was
  `flutter_inappwebview`, **swapped out — it doesn't build on this project's AGP 9.0.1**, ProGuard
  break). Loads `music.youtube.com` with a mobile UA; `onPageFinished` → `_tryCapture` reads the
  cookies natively (the auth cookie `__Secure-3PAPISID` is **httpOnly**, invisible to JS
  `document.cookie`), and once `__Secure-3PAPISID` is present pops `{cookie, sapisid}`.
- **Native** `getYtmCookies` (`MainActivity.kt`): returns
  `CookieManager.getInstance().getCookie("https://music.youtube.com")` (shared with the WebView).
- **`YouTubeService`** (`youtube_service.dart`): `isYtmLoggedIn`, `loadYtmAuth()` (from ctor,
  next to `_loadQueue`), `setYtmAuth(cookie, sapisid)`, `clearYtmAuth()` — cookie+SAPISID stored in
  **`flutter_secure_storage`** (Keystore). `_sapisidHash()` builds
  `SAPISIDHASH <ts>_<sha1(ts + " " + SAPISID + " " + "https://music.youtube.com")>` (`crypto`).
  `_innertubePost` (client stays WEB_REMIX) adds `Cookie` + `Authorization: <_sapisidHash()>` +
  `X-Goog-AuthUser: 0` when logged in, and **auto-`clearYtmAuth()` on 401/403** (silent revert to
  logged-out; anonymous album/artist/search paths are unchanged and keep working).

### Home feed → mix tiles → native queue (`youtube_service.dart`)
- **`HomeTile { title, thumbnail, playlistId?, songs? }`** + `List<HomeTile> homeTiles`.
- **`fetchHomeTiles()`** — browses `FEmusic_home`; collects `musicTwoRowItemRenderer` cards with a
  playable id (`watchPlaylistEndpoint.playlistId` RD…/RDCLAK5uy… else `browseEndpoint.browseId`
  VL…, filtered to `RD/VL/OLAK/PL`), plus `_quickPicksTile` (a `musicResponsiveListItemRenderer`
  song shelf → a `songs` tile). Selects a **stable set of 4** via `prefs = ['supermix','quick picks',
  'discover mix','new release mix','my mix','listen again']` (Supermix pinned first via `contains`),
  filling remaining slots in feed order. `[]`/keeps prior tiles on failure.
- **`getMixTracks(tile)`** — `tile.songs` if present, else `_playlistPanelTracks(pid)` (strips a
  `VL` prefix). **`_playlistPanelTracks(playlistId)`** was extracted from `getArtistRadioTracks`
  (`next` + `isAudioOnly:true` → `playlistPanelVideoRenderer` → tracks) — shared by radio + mixes.
- **`debugDumpHome()`** — TEMP Phase-1 validation hook (logs `FEmusic_home` shelves/cards). Can be
  removed once parsing is settled.

### UI
- **Favorites screen** (`main.dart` `_FavoritesSidebarState`): a **2×2 mix grid** on top
  (`GridView.count` of `_MixTile`s) then the favorites list in an `Expanded` below. Logged out → a
  single **"Connect YouTube Music"** button (pushes the login WebView → `setYtmAuth` +
  `fetchHomeTiles`); logged in + empty → skeletons + one-shot `fetchHomeTiles()`; tap a tile →
  snackbar → `getMixTracks` → `collab.loadQueueAndPlay` → `_goToQueueView()`.
- **Settings → Media & Collab**: `_YtMusicAccountCard` ("YouTube Music (personalized mixes)"),
  CONNECT pushes the WebView, DISCONNECT `clearYtmAuth`.
- **Deps added**: `webview_flutter`, `flutter_secure_storage`, `crypto`.

## ⚙️ Settings & feature flags (V3.5) — `DashboardProvider` + `lib/ui/settings_screen.dart`

Settings is a **fullscreen page**, not a dialog. It renders as a top-level `Positioned.fill`
layer in the dashboard `Stack` (same pattern as the welcome overlay), driven by
`DashboardProvider.showSettingsUI` + `forceShowSettingsUI()`/`dismissSettingsUI()`. The header
gear opens it; an **X** closes it. All colors come from `Theme.of(context)` so it tracks the
day/evening/night theme. `lib/ui/settings_dialog.dart` was **retired** (deleted).

### Feature-flag system (persisted, live)
Each optional feature has a **persisted flag** in `DashboardProvider` (SharedPreferences), loaded
by **`_loadSettings()` awaited at the top of `initialize()`** (before the `_start*` pollers).
Setter pattern: field + getter + `set*(bool)` that **persists → `notifyListeners()` → starts/stops
the underlying work**. Flags (keys):

| Flag / key | Default | Gates |
|---|---|---|
| `feat_speed_limit` | on | `_fetchSpeedLimit` call + the speed-limit sign (`speedometer_widget`) |
| `feat_speed_warning` | on | over-speed color + gauge redlines (`speedometer_widget`) |
| `feat_dashcam` | on | dashcam auto-start (`welcome_overlay`) + `_startDashcamPolling`/`stopDashcamPolling` + REC |
| `feat_welcome_overlay` | on | `checkNewDrive()` early-returns when off |
| `feat_maps_autolaunch` | on | START DRIVE → Maps + nav-wait (`welcome_overlay`) |
| `feat_notifications` | on | `notifications_tab` poll + list (provider-aware; shows a disabled placeholder) |
| `feat_share_favorites` | on | `initShareHandler()` + tears down `_shareSubscription` |
| `feat_adaptive_brightness` | on | persists the existing adaptive-brightness toggle |
| `phone_mode` (Int 0/1/2) | 1 | `_handleCallStateChange` early-returns when **0=Off**; 1=In-app only; 2=Full (default dialer) |

Collab reuses `CollabService`'s own `collab_enabled` (enable/disable + passenger sub-toggles); it
is **not** a `feat_*` flag.

### Missing-permission UX
A feature can be ON but **dormant**: consumers check `flag && permissionGranted`. The settings card
shows a **NEEDS PERMISSION** badge and a per-permission status row with a green check (Granted) or a
**Grant** button (no auto-prompt). Permission getters/requesters already on the provider:
`hasLocationPermission`/`checkLocationSettingsAndPermissions`, `hasNotificationAccess`/
`requestNotificationAccess`, `canDrawOverlays`/`requestOverlayPermission`, `hasPhonePermissions`/
`requestPhonePermissions`, `isDefaultDialer`/`requestDefaultDialer`. The screen refreshes status via
`checkPermissions()`/`checkPhonePermissions()` in a post-frame callback on open.

### Sections (`settings_screen.dart`)
Reusable `_Section`, `_FeatureCard` (Switch + optional permission rows + footnote), `_SegmentedControl`
(phone mode), `_PermStatus` (a plain class — call `.build(context)`). Seven sections: **Driving & Speed**,
**Media & Collab**, **Phone** (3-mode selector), **Notifications** (note: OS notification-access is shared
with media sync + dashcam, so turning tracking off keeps those working), **Display & System** (theme
override chips, adaptive brightness, draw-overlays), **Developer** (demo mode, simulate day/night),
**Permissions overview** (consolidated status/grant).

### V3.5 also did a lifecycle/concurrency cleanup pass
`DashboardProvider` now has a `_disposed` guard (`notifyListeners()` no-ops after dispose) + a full
`dispose()` that cancels every timer/stream and clears the platform handler; the media poll has a
single-timer + in-flight guard; brightness/ringer setters suppress the 3s poll briefly so it can't
revert a fresh user change; `YouTubeService.queueReady` (a `Completer`) lets `CollabService` reconcile
only after the queue restores; Auto-DJ has an `_advancing` re-entrancy flag.

## 📡 Passenger sharing: PWA ↔ backend ↔ dashboard

`backend/server.js` is a dumb relay keyed by `sessionId` (registered by the dashboard).
PWA is `backend/public/` (bump `app.js?v=N` in `index.html` when editing app.js — the SW
is pass-through, no caching). Key socket events:
- **Passenger → dashboard**: `join_passenger`, `request_queue`, `request_permissions`,
  `request_search` (`{query, source}`; source `ytmusic` default or `youtube` demo),
  `passenger_add_resolved` (exact YT Music pick — added directly, WYSIWYG),
  `passenger_add_song`/`add_song` (videoId, dashboard resolves), `passenger_play_song`
  (tap-to-play, gated on media permission), `passenger_media_action`
  (`playPause/next/previous`), `passenger_delete_song`, `passenger_reorder_song`.
- **Dashboard → passengers**: `update_queue`→`queue_updated`, `update_playing_status`→
  `now_playing_updated`, `update_play_state`→`play_state_updated` (drives the PWA
  play/pause icon), `update_permissions`, `update_media_permissions`, `search_results`.
- PWA niceties: crisp inline-SVG media/queue controls; queue greys out (opacity 0.5) after
  a media action until the dashboard confirms; tap-to-play rows when media control allowed.

The collab/queue view is `lib/ui/queue_tab.dart` — a thin view over `CollabService`:
default = queue list (QR never auto-shows; reachable via the QR icon), a highlighted
**COLLAB ON/OFF** toggle (= sharing), permission toggles, Clear Queue, and a **New Session**
button on the QR screen (the only full reset).
