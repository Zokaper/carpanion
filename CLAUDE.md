# Developer Guide (CLAUDE.md)

Carpanion is a Flutter **car dashboard / infotainment head-unit** app for an Android
tablet mounted in a vehicle. It started as a speedometer and grew into a full
infotainment hub: driving metrics, maps/nav automation, media control, notifications,
system controls, a **Collaborative Playback** feature (passengers add/edit the queue
from their phones via a PWA), and a **native music queue** driven off YouTube Music.

This file is the onboarding context — read it first when picking the project up on a
new machine.

---

## 🛠️ Build & Run

Uses **`puro`** for Flutter version management — always prefix Flutter commands with it.

* **Run**: `puro flutter run` (add `-d <device-id>` for a specific device)
* **Hot reload / restart**: `r` / `R` in the run terminal
* **Deps**: `puro flutter pub get`   •   **Clean**: `puro flutter clean`
* **Verify a change compiles (Dart + Kotlin)**: `puro flutter build apk --debug`
* **Analyze**: `puro flutter analyze` — NOTE: `lib/main_corrupted.dart` is a dead,
  broken backup file that throws ~30 analyzer errors; ignore it (it is not imported/
  compiled). There are also many pre-existing `withOpacity` deprecation **info** lints —
  those are noise, not errors.

### Backend (PWA + WebSocket relay)
* **Path**: `backend/`   •   **Start**: `node server.js`
* **Hosted**: the dashboard app and passenger phones both connect to the **remote**
  backend at `https://carpanion.onrender.com` (hardcoded in `collab_service.dart` and
  `queue_tab.dart`). Render **auto-deploys `backend/` on push to `main`**. So PWA/server
  changes only reach phones after a push to main; Flutter/native changes need an app rebuild.

### Git workflow
* Active branch is **`V3`**; `main` tracks it. **Push to BOTH** every time:
  `git push origin V3`, then `git checkout main && git merge --ff-only V3 && git push origin main && git checkout V3`.
* When staging, **exclude** `macos/Flutter/GeneratedPluginRegistrant.swift` (regenerated by
  `pub get`, not real work).

## 🏗️ Architecture Overview

Three cooperating layers:

1. **Flutter UI + state** (`lib/`) — dashboard, media panel, favorites, collab/queue tab.
   State lives in `ChangeNotifier` providers wired in `main.dart`'s `MultiProvider`:
   `DashboardProvider`, `DynamicThemeProvider`, `YouTubeService`, `CollabService`
   (the last is created after the others so it can `context.read` them).
2. **Native Android** (`android/.../MainActivity.kt` + `DashcamListenerService.kt`) —
   platform channel `com.example.car_dashboard/system`; drives YT Music via its
   MediaSession, reads media metadata, dashcam/brightness/ringer/network, notification
   listening.
3. **Node backend + PWA** (`backend/`) — `server.js` is a socket.io **relay** between the
   dashboard and passenger phones; `backend/public/` is the passenger PWA
   (`index.html`, `app.js`, `sw.js`). Passengers join by scanning a QR shown on the tablet.

## 🎵 The Core Idea: one native "Now Playing Queue"

The pivotal capability: we can drive YouTube Music **without flashing it to the
foreground** by sending commands to its MediaSession. Native handler
**`playFromMediaSession`** (`MainActivity.kt`) calls `transportControls.playFromUri`
with `https://music.youtube.com/watch?v=<videoId>` — confirmed to play silently in the
background (`playFromSearch` is silently ignored by YT Music, so we always use URI).

On top of that we built a **single app-managed queue** — the app owns the list, plays it
one track at a time via `playFromMediaSession`, and auto-advances. YT Music is just the
audio backend. This queue powers **both** Collab (passenger adds) and **Favorites**
(albums/artists/songs). Everything funnels through **`CollabService`**.

### `CollabService` (`lib/services/collab_service.dart`) — the playback engine
App-lifetime service (survives navigation; the collab/queue tab is a thin view over it).
- **Queue** lives in `YouTubeService.currentQueue` (a `List<Map>` of `{id, videoId,
  originalVideoId, title, artist, thumbnail}`), persisted to `SharedPreferences`
  (`collab_queue`).
- **`playAt(index)`** → sets `_currentPlayingIndex`, `_playbackActive = true`, invokes
  `playFromMediaSession`; falls back to a launch intent if the MediaSession is unreachable.
- **Auto-DJ** (`_onDashboardUpdate`, listens to `DashboardProvider`): when the playing
  track (from native polling) leaves the queue (song ended → YT autoplay), it advances to
  the next queue item. **Gated on `_playbackActive`, NOT on collab being enabled** — so
  favorites playback auto-advances even with Collab off. It also re-syncs the highlighted
  index by fuzzy title-match and emits `update_playing_status` / `update_play_state`.
- **Favorites entry points**: `playFavoriteSong({videoId,title,artist,thumbnail})` and
  `loadQueueAndPlay(List<songs>)` — both **replace** the queue and play from 0.
- **Persistence** (`SharedPreferences`): `collab_session_id`, `collab_enabled`,
  `collab_index`. On restart it does a **silent restore**: re-selects the saved song
  (matching what YT Music is already playing) but does NOT replay it (would restart the
  song at 0:00); resumes Auto-DJ if a match is found.
- **Media controls**: `next()`/`previous()` (queue-aware, else native prev/next).

### Sharing-layer Collab model (IMPORTANT semantics)
There is ONE queue. **`_enabled` = "passenger sharing is open", NOT playback.**
- Toggling Collab ON/OFF **never** starts, stops, or clears playback — flip it mid-album
  and the music is untouched. Enabling just opens the door for passengers.
- Passenger **write** actions (`passenger_*` socket handlers) are gated on `_enabled`
  (and the per-permission toggles `_allowEditing` / `_allowMediaControl`).
- Playback starts from: a favorite (song/album/artist), tap-to-play a queue row, or a
  passenger add to an idle queue (`_maybeAutoStart`). **Enabling Collab does NOT auto-play.**
- `newSession()` (the only full reset: new session id + clear queue) and `clearQueue()`
  (empties queue, keeps session) both reset `_playbackActive`/index.

## 🔎 YouTube Music integration (`lib/services/youtube_service.dart`)

- **OAuth** via `google_sign_in` (youtube scope) → `YouTubeApi` (Data API v3). Used for
  the demo/"search YouTube" path (`searchSongs`) and fetching a video's real title.
- **Innertube (WEB_REMIX, anonymous)** — the good stuff. Public API key, POST to
  `music.youtube.com/youtubei/v1/...`:
  - **`searchYTMusicSongs(query, {limit})`** — "Songs" filter
    (`params: 'EgWKAQIIAWoKEAkQBRAKEAMQBA=='`). Returns ranked `{videoId,title,artist,
    thumbnail}` list via `_parseSongRenderer`. This is the authoritative song resolver AND
    the source of the PWA's search results (replaced iTunes, which ranked badly).
  - **`getAlbumTracks(album, artist)`** — `search` → find `MPRE…` album browseId →
    `browse` → collect `musicResponsiveListItemRenderer` tracks (fills missing artist/
    thumbnail from album context).
  - **`getArtistRadioTracks(artist)`** — `search` → `UC…` artist browseId → `browse` →
    find an `RD…` radio playlistId → `next` → parse `playlistPanelVideoRenderer`. Gives
    YT Music's own popularity-weighted shuffle order.
  - Generic recursive JSON walkers: `_findBrowseId`, `_findRadioPlaylistId`,
    `_findFirstThumbnail`, `_collectByKey`, `_runsText`; shared POST via `_innertubePost`.
  - **All fetchers return `[]` on failure** so callers fall back to the legacy YT Music
    launch. They have debug logging (`getAlbumTracks: … → N tracks`, etc.).
- **`replaceQueue(songs)`** / **`addResolvedSong(...)`** / `addVideoToPlaylist` / dedup +
  5s repeat guard, all persisting the queue.

> ⚠️ The album/artist Innertube parsing was written **without live-response validation** —
> it's the main risk area. If album/artist playback falls back to the YT flash, check the
> debug logs and adjust the parse. Search + single-song resolution are proven/reliable.

## ⭐ Favorites (typed) — `main.dart` `DashboardProvider` + `_MediaControlPanelState` + `_FavoritesSidebarState`

Favorites are now **typed** maps: `type ∈ {song, album, artist, playlist}`.
- `song` → stores resolved `videoId` + `thumbnail` (resolved via `searchYTMusicSongs` at
  favorite time so playback is instant).
- `album`/`artist` → store name(s); tracks are fetched on play.
- `playlist` → legacy/dynamic (incl. the default "My Supermix" etc.); launches YT Music
  via the old intent (`_launchPlaylist`).
- **Persistence**: single JSON list under `fav_json` (migrates the old parallel
  `fav_titles/urls/subtitles` keys on first load; `type` defaults to `playlist`).
- **Media-player buttons** (top-right of the now-playing panel, `_MediaControlPanelState`):
  heart = song (`_toggleSongFavorite` resolves the id), `Icons.album` = album (needs
  `_currentAlbum`), `Icons.interpreter_mode` = artist. Toggles via `DashboardProvider`
  helpers `isSongFavorited/isAlbumFavorited/isArtistFavorited`, `addSongFavorite`,
  `toggleAlbumFavorite`, `toggleArtistFavorite`.
- **`_currentAlbum`** is populated by polling the native **`getCurrentMediaMetadata`**
  (title/artist/album from the YT Music MediaSession) in `DashboardProvider._startMediaPolling`
  — the media plugin only exposes track/artist.
- **Tapping a favorite** → `_playFavorite` dispatcher: song→`playFavoriteSong`;
  album→`getAlbumTracks`→`loadQueueAndPlay`; artist→`getArtistRadioTracks`→
  `loadQueueAndPlay`; playlist / any failed fetch → `_launchPlaylist` (legacy YT launch).

## 📡 Passenger sharing: PWA ↔ backend ↔ dashboard

`backend/server.js` is a dumb relay keyed by `sessionId` (registered by the dashboard).
PWA is `backend/public/` (bump `app.js?v=N` in `index.html` when editing app.js — the SW
is pass-through, no caching). Key socket events:
- **Passenger → dashboard**: `join_passenger`, `request_queue`, `request_permissions`,
  `request_search` (`{query, source}`; source `ytmusic` default or `youtube` demo),
  `passenger_add_resolved` (exact YT Music pick — added directly, WYSIWYG),
  `passenger_add_song`/`add_song` (videoId, dashboard resolves), `passenger_play_song`
  (tap-to-play, gated on media permission), `passenger_media_action`
  (`playPause/next/previous`), `passenger_delete_song`, `passenger_reorder_song`.
- **Dashboard → passengers**: `update_queue`→`queue_updated`, `update_playing_status`→
  `now_playing_updated`, `update_play_state`→`play_state_updated` (drives the PWA
  play/pause icon), `update_permissions`, `update_media_permissions`, `search_results`.
- PWA niceties: crisp inline-SVG media/queue controls; queue greys out (opacity 0.5) after
  a media action until the dashboard confirms; tap-to-play rows when media control allowed.

The collab/queue view is `lib/ui/queue_tab.dart` — a thin view over `CollabService`:
default = queue list (QR never auto-shows; reachable via the QR icon), a highlighted
**COLLAB ON/OFF** toggle (= sharing), permission toggles, Clear Queue, and a **New Session**
button on the QR screen (the only full reset).

## 🚧 Known risks & next steps
- **Album/artist Innertube parsing** needs on-device validation against real responses
  (see the warning above). Falls back to YT Music launch on failure.
- **Supermix / dynamic personal playlists**: deferred. Our OAuth token drives the Data API
  (which doesn't expose YT Music mixes); Innertube auth would need a separate TV/limited-
  input device flow (how `ytmusicapi` does it). For now they're `playlist` favorites that
  launch YT Music. A future spike: try OAuth-bearer Innertube with an `ANDROID_MUSIC`/`TV`
  client to fetch mixes natively.
- **Playing highlight** uses fuzzy title-matching (native track title vs stored queue
  title) — unusual titles can occasionally mis-highlight.
- Passenger edit/media **permissions reset to off on app restart** (safe default; not persisted).
